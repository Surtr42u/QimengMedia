# AGENTS - 绮梦影库 AI 代理规则

本文件面向所有 AI 编码代理。修改任何代码或文档前，必须先读取 `AI_README_FIRST.md`。

## 强制流程

1. 先读 `AI_README_FIRST.md`（区分首次接手还是日常任务）。
2. **再读 `docs/GUIDE_CODE_MAINTENANCE.md`**（代码规范第一要务，必须在功能指南之前阅读。包含共享逻辑模式、Fragment 拆分规范、编码规范等，是"怎么写代码"的前提）。
3. 再读 `docs/GUIDE_USAGE_MATRIX.md`（匹配触发词）。
4. 根据任务类型读取对应外部 Skill、`docs/AI_WRITTEN_SKILLS/*.md` 和 `docs/GUIDE_*.md`。
5. 基于文档中的文件职责、布局描述、数据规则直接定位并修改代码，**不全文通读源码**。
6. **涉及日志诊断时，必须先读 `docs/GUIDE_DEBUG.md`**，使用 AppLog 文件日志（`run-as` 方式），禁止使用 `adb logcat`（Android 16 返回空缓冲区）。

程序员初次接手建议先看 `docs/GUIDE_README.md` 的《你想做什么？》快速导航。

## 主动追问原则

用户说"把这个界面改一下"、"感觉有点卡"等模糊请求时，AI 不能直接猜测意图就开始写代码。必须：

1. **先确认范围**：用户说的是哪个页面？是整个页面还是某个组件？
2. **追问细节**：改什么方向？变大变小？换颜色？加按钮？删东西？
3. **补充遗漏**：用户可能不知道会影响到什么——AI 读完指南后应主动提示"这个改动会涉及 XX 模块、需要更新 XX 文档"
4. **涉及删除时**：必须确认"UI 入口删了，底层代码我也要一起清理吗？"

做一个追问型助手，不要做一个猜谜型助手。

## 源码隔离原则

- 理解项目只能通过 `docs/PROJECT_GUIDE.md` 和 `docs/GUIDE_*.md`，禁止通读源码来了解项目。
- 修改代码时，只读取目标文件的**相关片段**（方法签名、变量定义、逻辑段），禁止全文通读。
- 如果文档未覆盖某段代码，读取后必须先更新文档再修改代码。
- 如果发现文档与代码不一致，先向用户确认正确版本，再统一。

## 文档维护

修改代码后必须同步更新文档，映射关系参见 `AI_README_FIRST.md` 中的《文档维护规则》表格。

核心规则：
- 文件增删改 → `docs/PROJECT_GUIDE.md`
- 新增功能模块 → 按 `AI_README_FIRST.md`《新增功能模块的文档流程》执行（7 步，模板见 `docs/_GUIDE_TEMPLATE.md`）
- 页面/UI 变更 → `docs/GUIDE_UI.md`
- 数据层变更 → `docs/GUIDE_DATA.md`（含 JSON 格式定义）；如导入导出流程变更，同步 `docs/GUIDE_BACKUP.md`
- 算法/规则变更 → 对应 `docs/GUIDE_*.md`
- 更新不及时的文档比没有文档更危险

文档描述质量约束：
- **独立性必须标注**：新增与现有功能有相似概念但独立的数据/实体/模块时，必须明确写清"与 XX 完全独立"及差异（不同实体/表/DAO/备份/粒度等），避免后续 AI 或开发者混淆
- **关键设计决策必须记录**：为什么这样设计而非另一种方案，尤其是涉及"两套独立系统""共用 vs 独立""为什么不用同一个表"等容易产生疑问的决策
- **数据边界必须明确**：每个实体的职责边界、与其他实体的关系（关联/独立/互斥）必须写清，不能只写实体名和字段列表
- **文档职责边界必须清晰**：每个 `GUIDE_*.md` 只写自己模块的内容，不写属于其他模块的细节。需要引用其他模块时用 `详见 GUIDE_XXX.md「章节名」`，不内联展开。`## 职责` 中的 `不管：` 声明必须明确排除范围并指向对应指南

## 删除功能的强制规则（极其重要）

**删除一个功能或 UI 入口时，禁止只删 UI 层而保留底层代码和文档。** 必须同时完成三件事：

1. **删代码**：从 UI → ViewModel → Repository 接口 → Repository 实现 → DAO（如无其他调用者）逐层删除，不留死代码
2. **删文档**：检查并更新 `docs/PROJECT_GUIDE.md`、对应的 `docs/GUIDE_*.md`、`docs/DEVELOPMENT_PLAN.md` 中所有对该功能的描述
3. **验证**：确认无其他代码引用被删除的方法，确认文档中无残留描述

如果没有逐层清理干净，就会造成"UI 已删但底层代码还在、文档还在说过时的规则"的状态——这是不可接受的。

## 回复签名（强制）

**每次完成代码修改任务后，必须在回复末尾附上本次实际读取过的文档清单。** 格式要求：

- 回复末尾另起一行，以 `📋` 开头
- 列出**本次确实读取过的**文档文件名（不含路径前缀）
- 未读取的文档不得列入

示例格式：
```
📋 已读取文档：GUIDE_UI.md, GUIDE_THEME.md, GUIDE_ANIMATION.md
```

或含外部 Skill 时：
```
📋 已读取文档：rshzrh-ui/SKILL.md, ANDROID_UI_DESIGN_SKILL.md, GUIDE_UI.md, GUIDE_THEME.md
```

**这一行是用户判断项目文档体系是否起效的唯一手段。如果回复末尾没有 `📋` 签名行，说明 AI 跳过了文档，用户应要求其补读。**

纯对话、纯回答问题、不做代码修改时，不需要签名。

## git 提交约束（强制）

修改代码时，**代码和文档必须在同一个 commit 中**。不允许代码和文档分两次提交。

commit message 格式：`类型(模块): 简述 | 文档: 已更新XXX`

示例：
```
fix(data): 删除死代码resetStats/clearUsageData | 文档: GUIDE_DATA.md, PROJECT_GUIDE.md
feat(ui): 详情页加倍速按钮 | 文档: GUIDE_UI.md, GUIDE_THEME.md
```

**多 AI 协作**：提交前必须 `git pull`。如果发现自己的修改会覆盖别人的文档改动，先合并再提交。不应该出现两个 commit 交替修改同一个文件。

## 工具调用增强原则

AI 在编写功能、修复 Bug、设计方案时，**必须积极调用可用工具**获取外部信息和参考，禁止闭门造车。具体要求：

1. **联网搜索**：遇到不熟悉的 API、框架用法、平台兼容性问题、性能优化方案时，必须使用 WebSearch 搜索权威来源（Android 官方文档、Kotlin 官方文档、GitHub 高星项目 Issue/讨论、StackOverflow 高票回答），禁止凭模糊记忆编写代码。
2. **网页抓取**：搜索到相关文档或讨论后，使用 WebFetch 抓取具体内容，确认 API 签名、参数、行为与当前项目目标一致后再采用。
3. **信息源质量把控**：
   - 优先使用：Android Developers 官方文档、Kotlin 官方文档、GitHub 官方仓库 Issue/Release、Jetpack 组件源码注释
   - 可参考：StackOverflow 高票回答（验证过被采纳的）、Medium/Dev.to 知名开发者文章
   - 禁止使用：来路不明的博客复制粘贴站、无出处代码片段、未标明版本的过时教程、可能植入恶意代码的第三方下载站
4. **外部代码仅作参考**：从搜索结果中获取的代码片段只能作为理解思路和 API 用法的参考，**禁止照搬**。必须根据项目现有架构、命名规范、数据模型进行适配重写，确保与项目代码风格一致。
5. **方案对比**：当存在多种实现方案时，应搜索对比各方案的优缺点、性能差异、兼容性风险，向用户说明后选择最优方案，而非随意选一种就写。
6. **Bug 修复参考**：遇到非显而易见的 Bug 时，应搜索该问题的已知原因和社区解决方案，避免在错误方向上反复尝试。

### 搜索频率和质量约束

1. **搜索前先自查**：调用 WebSearch 前先确认：
   - 项目指南文档是否已覆盖此问题？
   - 是否能用 Grep/Read 在项目代码中找到答案？
   - 是否属于自己熟悉的 Android/Kotlin API？
   - 以上任一为"是"则不搜索

2. **搜索频率限制**：
   - 同一任务中，同一主题最多搜索 2 次
   - 搜索后优先抓取官方文档（developer.android.com、kotlinlang.org）
   - 禁止连续搜索同一问题的不同表述

3. **搜索结果过滤**：
   - 只抓取排名前 3 的搜索结果
   - 优先抓取：官方文档 > GitHub Issue > StackOverflow 高票回答
   - 禁止抓取：复制粘贴站、无出处博客、未标明版本的教程
   - 抓取后只提取关键信息（API 签名、参数、行为），禁止照搬代码

4. **WebFetch 约束**：
   - 每次 WebSearch 后最多抓取 2 个页面
   - 抓取前先看搜索结果摘要，判断是否值得抓取
   - 抓取后只提取与当前任务直接相关的信息

5. **禁止盲目搜索的场景**：
   - 项目已有指南文档覆盖的问题
   - 标准 Kotlin/Android API 用法（应直接使用）
   - 已在之前的对话中搜索过的问题
   - 纯逻辑/算法问题（应独立思考）

## 代码自审（强制）

**每次完成代码修改、准备向用户回复之前，AI 必须对本次所有代码变更进行一次自审。** 自审清单：

1. **逻辑正确性**：修改是否真正解决了问题？是否存在边界条件遗漏？是否有空指针/越界风险？
2. **线程安全**：数据库操作是否在 `Dispatchers.IO`？UI 更新是否在主线程？是否有并发竞态？
3. **性能影响**：是否引入了全量加载（`getAll()`）？是否在主线程做了耗时操作？大数据集下是否会 OOM？
4. **一致性**：修改是否与项目现有架构一致（ViewModel → Repository → DAO）？命名风格是否统一？
5. **副作用**：修改是否影响了其他模块的功能？是否需要同步更新其他文件？
6. **文档同步**：是否已更新对应的 `docs/GUIDE_*.md` 和 `docs/PROJECT_GUIDE.md`？

如果自审发现问题，**必须先修复再回复用户**，不得将已知问题留给用户发现。

## 构建验证规则

AI 运行 `gradlew assembleDebug` 等构建命令后，**必须执行 `gradlew --stop` 停止 Gradle daemon**。原因：Windows 上命令行 Gradle daemon 会持有文件锁，导致 Android Studio 点击运行时报错（daemon 冲突、构建锁冲突）。停止 daemon 不影响下次构建速度（Android Studio 会自动启动自己的 daemon）。

## 禁止行为

- 禁止跳过指南直接写代码。
- 禁止修改代码后回复末尾不加 `📋` 已读取文档签名。
- 禁止在签名中列入未实际读取的文档。
- 禁止通读完整源码文件来理解项目。
- 禁止修改 `app/ai-skills/external/` 下外部 Skill 原文，除非用户明确要求。
- 禁止新增功能后不更新项目文档。
- 禁止删除功能时只删 UI 层而保留底层实现代码和文档。
- 禁止修改、删除、移动用户原始媒体文件。
- 禁止凭模糊记忆编写不熟悉的 API/框架代码，必须先搜索确认。
- 禁止照搬外部搜索到的代码片段，必须适配项目架构后重写。
- 禁止使用低质量信息源（复制粘贴站、无出处代码、未标明版本的教程）。
- 禁止完成代码修改后跳过自审直接回复用户。

## 冲突优先级

1. 用户最新要求。
2. `docs/PROJECT_GUIDE.md`。
3. `docs/GUIDE_*.md`。
4. `docs/GUIDE_USAGE_MATRIX.md`（触发规则定义）。
5. `docs/AI_WRITTEN_SKILLS/*.md`。
6. `app/ai-skills/external/*`。

如果发现自己没有读取当前任务对应指南，必须停止实现并补读。
