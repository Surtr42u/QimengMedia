# 通用项目文档框架 - AI 自举指南

本文档描述了如何为一个新项目（无论平台/语言）快速建立完整的 AI 开发文档体系。任何一个 AI 代理阅读本文档后，应该能在 **10 分钟内**为新项目生成一个与绮梦影库同等质量的文档骨架。

---

## 一、文档体系的核心理念

### 为什么需要这套体系

- **源码隔离**：AI 通过文档理解项目，不全文通读源码（防 token 爆炸）
- **上下文复用**：AI 每次只读命中模块的指南，token 消耗可控
- **强制同步**：代码和文档必须同一 commit 提交，不一致的文档比没有更危险
- **可验证性**：回复末尾的 `📋` 签名让用户一眼看出 AI 有没有读文档

### 三层指南结构

| 层级 | 位置 | 性质 | 修改权限 |
|---|---|---|---|
| 外部 Skill | 项目目录下的 `external/` | 第三方通用规范，下载自 GitHub | 只读 |
| 自写通用 Skill | `docs/AI_WRITTEN_SKILLS/` | 本项目特有的工程约束（精简版，不重复外部 Skill 和 GUIDE） | 可维护 |
| 项目专属指南 | `docs/GUIDE_*.md` | 本项目专属指南 | 必须与代码同步 |

### 自写 Skill 精简原则

自写 Skill **只保留本项目特有的工程约束**（B 类），不重复以下内容：

- **A 类（通用知识）**：外部 Skill 已覆盖的通用知识，不重复写。用"通用 XX 知识见外部 Skill `xxx`"指向。
- **C 类（GUIDE 已有）**：项目专属业务规则已在 `docs/GUIDE_*.md` 中描述，不重复写。用"详见 GUIDE_XXX.md"指向。

B 类约束同时存在于自写 Skill 和对应 GUIDE 的「修改注意事项」中，修改时必须同步两处。

## 二、必须创建的文件清单

### 根目录（3 个）

| 文件 | 角色 |
|---|---|
| `README.md` | 人类/AI 第一入口：项目简介、技术基准、核心规则 |
| `AI_README_FIRST.md` | **唯一权威规则入口**：读取顺序、源码隔离、签名、文档维护、新增/删除流程、git 约束、构建验证、禁止行为、冲突优先级、工具调用、代码自审 |
| `AGENTS.md` | AI 代理快速索引：指针引用 AI_README_FIRST + 仅收录独有内容（git 提交约束、构建验证规则、禁止行为、冲突优先级） |

### docs/ 目录（最少 6 个）

| 文件 | 角色 |
|---|---|
| `PROJECT_GUIDE.md` | 项目全貌：定位、基准、实现进度（结论性条目）、目录结构、核心原则 |
| `GUIDE_README.md` | 指南索引 + 人类快速导航《你想做什么？》 |
| `GUIDE_USAGE_MATRIX.md` | 任务×指南触发矩阵（含自然语言匹配） |
| `DEVELOPMENT_PLAN.md` | 分阶段开发计划 + 统一待办清单 + 技术债记录 |
| `CHANGELOG.md` | 历次代码审查、重构、性能优化的归档记录（从 GUIDE_CODE_MAINTENANCE 剥离的历史表） |
| `_GUIDE_TEMPLATE.md` | 新建指南时的可复制模板 |

### 项目专属指南（按需，最少 3 个）

`docs/GUIDE_UI.md` — UI/页面  
`docs/GUIDE_DATA.md` — 数据层  
`docs/GUIDE_FILE_NAME.md` — 根据项目实际模块命名

### 自写通用 Skill（按需）

`docs/AI_WRITTEN_SKILLS/README.md` — 自写 Skill 索引 + 强制同步映射表

具体文件按项目技术栈决定（见下面决策树）。

## 三、决策树：根据项目类型选择要创建什么

### 第一步：确认平台和技术栈

| 问题 | 示例回答 |
|---|---|
| 什么平台？ | Android / iOS / Web / 桌面 / CLI |
| 什么语言？ | Kotlin / Swift / TypeScript / Python / Rust |
| 什么架构？ | MVVM / MVC / Redux / Clean Arch |
| 有 UI 吗？ | 有（原生/CSS/Compose）/ 无（纯 CLI/后端） |
| 有数据库吗？ | 有（SQLite/Postgres/etc.）/ 无 |
| 需要备份/导入导出吗？ | 是 / 否 |
| 有算法/推荐吗？ | 是 / 否 |

### 第二步：按技术栈选自写 Skill

| 技术栈 | 必须创建的自写 Skill |
|---|---|
| Android + Kotlin + MVVM | `ANDROID_ARCHITECTURE_SKILL.md`、`ANDROID_VIEW_SKILL.md` |
| Android + Room | 追加 `ANDROID_ROOM_SKILL.md` |
| Android + SAF | 追加 `ANDROID_STORAGE_SAF_SKILL.md` |
| Android + 媒体 | 追加 `ANDROID_MEDIA_SKILL.md` |
| 任何平台 + 图片/视频 | 平台相关的媒体处理 Skill |
| 任何平台 + 数据库 | 平台相关的数据层 Skill |
| 任何平台 + 多主题 | 创建 `GUIDE_THEME.md`（项目专属） |
| 任何平台 + 性能敏感 | 创建 `PERFORMANCE_SKILL.md` |

### 第三步：按功能选项目指南

| 功能 | 创建的项目指南 |
|---|---|
| 有页面/UI | `GUIDE_UI.md` |
| 有数据/存储 | `GUIDE_DATA.md` |
| 有多主题 | `GUIDE_THEME.md` |
| 有扫描/文件访问 | `GUIDE_SCAN.md` |
| 有推荐/排序算法 | `GUIDE_ALGORITHM.md` |
| 有备份/导入导出 | `GUIDE_BACKUP.md` |
| 有用户管理系统 | `GUIDE_USER.md`（或对应名称） |
| 有动画/动效 | `GUIDE_ANIMATION.md` |
| 有平台兼容要求 | `GUIDE_COMPAT.md` |
| 有调试/日志需求 | `GUIDE_DEBUG.md` |
| 有代码维护/重构需求 | `GUIDE_CODE_MAINTENANCE.md` |

### 4.1 README.md 模板

```markdown

4. `docs/GUIDE_CODE_MAINTENANCE.md`（代码规范第一要务）

3. 改代码 + 同步文档 + 回复末尾 `📋` 签名

## 核心规则

- [项目特有的不可违反约束 1]
- [项目特有的不可违反约束 2]
- 外部 Skill 只读，禁止修改
- AI 自写规范可维护，但必须与对应 GUIDE_* 同步（见 AI_WRITTEN_SKILLS/README.md 映射表）
- 项目专属指南必须与代码同步
- 触发矩阵是任务→指南的唯一权威来源
- 项目交接必须传完整源码

- 外部搜索到的代码仅作参考，禁止照搬，必须适配项目架构后重写
- 信息源质量把控：优先官方文档，禁止使用低质复制粘贴站
- 每次完成代码修改后必须自审，发现问题先修复再回复用户

## 当前基准

- [功能 3]
```

### 4.2 AI_README_FIRST.md 模板

```markdown
# AI_README_FIRST - AI 接手强制阅读规则

任何 AI、代码助手或开发者在修改本项目前，必须先读本文件。

## 你是哪种情况？

### 首次接手项目
按《总读取顺序》完整走一遍，建好全局认知。

### 日常任务
1. 打开 `docs/GUIDE_USAGE_MATRIX.md`，匹配用户请求的触发词
2. 只读命中的指南和文档
3. 基于文档定位代码片段修改
4. 改完后更新对应文档 + 回复末尾加 `📋` 签名

## 总读取顺序（首次接手用）

1. `README.md`
2. `AI_README_FIRST.md`
3. `AGENTS.md`
4. `docs/GUIDE_CODE_MAINTENANCE.md`（代码规范第一要务，必须在功能指南之前阅读）
5. `docs/GUIDE_USAGE_MATRIX.md`
6. [外部 Skill 索引文件]
7. [外部 Skill 原文（如有）]
8. `docs/AI_WRITTEN_SKILLS/README.md`
9. 与任务相关的 `docs/AI_WRITTEN_SKILLS/*.md`
10. `docs/PROJECT_GUIDE.md`
11. `docs/GUIDE_README.md`
12. 与任务相关的 `docs/GUIDE_*.md`
13. 对应源码（仅限本文档允许的片段读取，禁止全文通读）

## 源码隔离原则

AI 理解项目时，禁止直接通读完整源码文件。必须通过文档体系理解项目。

仅在以下情况允许读源码：
- 修改具体实现细节（方法签名、变量名、逻辑修正）
- 文档未覆盖的代码段，且只读目标相关片段
- 读取后发现文档未记录的信息，必须先更新文档再改代码

## 回复签名（强制）

每次完成代码修改后，回复末尾必须签上实际读取过的文档清单：
格式：`📋 已读取文档：[文件名列表]`
纯对话不需要签名。

## 文档维护规则

| 代码变更类型 | 必须同步的文档 |
|---|---|
| 新增/删除/移动文件 | `PROJECT_GUIDE.md` |
| 新增功能模块 | 按本文件《新增功能模块的文档流程》执行 |
| 修改页面/UI | `GUIDE_UI.md` |
| 修改数据实体/策略 | `GUIDE_DATA.md`（含备份格式） |
| 修改调试/日志 | `GUIDE_DEBUG.md` |
| [其他按项目实际模块扩展...] |

## 删除功能的强制规则

删除功能：UI → 逻辑层 → 数据层 → DAO 逐层清理代码
同时清除 PROJECT_GUIDE、对应 GUIDE_*、DEVELOPMENT_PLAN 中的描述

## 工具调用增强原则

AI 代理在实现功能、修复 Bug、设计方案时，必须积极调用可用工具（联网搜索、网页抓取等）获取外部信息，**禁止闭门造车**。

### 何时必须调用工具

| 场景 | 应调用的工具 | 目的 |
|---|---|---|
| 不熟悉的 API / 框架用法 | WebSearch → WebFetch | 确认 API 签名、参数、行为 |
| 平台兼容性问题 | WebSearch | 查找官方兼容性说明和已知 Issue |
| 性能优化方案选择 | WebSearch | 对比多种方案的性能数据和社区反馈 |
| 非显而易见的 Bug | WebSearch | 搜索已知原因和社区解决方案 |
| 多种实现方案可选 | WebSearch + WebFetch | 对比优缺点后选择最优方案 |
| 第三方库升级 / 新依赖引入 | WebSearch → WebFetch | 确认版本兼容性、迁移指南 |

### 信息源质量分级

| 等级 | 来源 | 使用方式 |
|---|---|---|
| **优先使用** | 官方文档、GitHub 官方仓库 Issue/Release、框架源码注释 | 直接参考，高可信度 |
| **可参考** | StackOverflow 高票回答（被采纳的）、知名开发者文章 | 交叉验证后采用 |
| **禁止使用** | 来路不明的博客复制粘贴站、无出处代码片段、未标明版本的过时教程、可能植入恶意代码的第三方下载站 | 不得采纳 |

### 外部代码使用规则

从搜索结果中获取的代码片段**只能作为理解思路和 API 用法的参考**，禁止照搬。必须：
1. 根据项目现有架构进行适配
2. 遵循项目命名规范和数据模型
3. 确保与项目代码风格一致
4. 不得引入项目未使用的第三方依赖

### 搜索频率和质量约束

1. **搜索前先自查**：调用 WebSearch 前先确认：
   - 项目指南文档是否已覆盖此问题？
   - 是否能用 Grep/Read 在项目代码中找到答案？
   - 是否属于自己熟悉的 API/框架？
   - 以上任一为"是"则不搜索

2. **搜索频率限制**：
   - 同一任务中，同一主题最多搜索 2 次
   - 搜索后优先抓取官方文档
   - 禁止连续搜索同一问题的不同表述

3. **搜索结果过滤**：
   - 只抓取排名前 3 的搜索结果
   - 优先抓取：官方文档 > GitHub Issue > StackOverflow 高票回答
   - 禁止抓取：复制粘贴站、无出处博客、未标明版本的教程
   - 抓取后只提取关键信息，禁止照搬代码

4. **WebFetch 约束**：
   - 每次 WebSearch 后最多抓取 2 个页面
   - 抓取前先看搜索结果摘要，判断是否值得抓取
   - 抓取后只提取与当前任务直接相关的信息

5. **禁止盲目搜索的场景**：
   - 项目已有指南文档覆盖的问题
   - 标准 API 用法（应直接使用）
   - 已在之前的对话中搜索过的问题
   - 纯逻辑/算法问题（应独立思考）

## 代码自审（强制）

**每次完成代码修改、准备向用户回复之前，AI 必须对本次所有代码变更进行一次自审。** 这是防止低级错误流入用户设备的最后防线。

### 自审清单

| 审查项 | 检查内容 |
|---|---|
| **逻辑正确性** | 修改是否真正解决了问题？边界条件是否遗漏？空指针/越界风险？ |
| **线程安全** | 数据库/IO 操作是否在后台线程？UI 更新是否在主线程？并发竞态？ |
| **性能影响** | 是否引入了全量加载？主线程耗时操作？大数据集 OOM 风险？ |
| **架构一致性** | 是否遵循项目分层架构？命名风格是否统一？ |
| **副作用** | 修改是否影响了其他模块？是否需要同步更新其他文件？ |
| **文档同步** | 是否已更新对应的项目文档？ |

### 自审结果处理

- 自审发现问题 → **必须先修复再回复用户**
- 不得将已知问题留给用户发现
- 如果修复引入新问题，重新走自审流程

## 新增功能模块的文档流程

新功能不属于任何已有指南时：
1. 创建 `GUIDE_XXX.md`（参照 `_GUIDE_TEMPLATE.md`）
2. 如涉及新技术，创建 `AI_WRITTEN_SKILLS/NEW_SKILL.md`
3. 更新 `PROJECT_GUIDE.md`
4. 更新 `GUIDE_USAGE_MATRIX.md`
5. 更新 `GUIDE_README.md`
6. 更新本文档《文档维护规则》表格
7. 更新 `DEVELOPMENT_PLAN.md`

## git 提交约束（强制）

修改代码时，**代码和文档必须在同一个 commit 中**。不允许代码和文档分两次提交。

commit message 格式：`类型(模块): 简述 | 文档: 已更新XXX`

**多 AI 协作**：提交前必须 `git pull`。如果发现自己的修改会覆盖别人的文档改动，先合并再提交。

## 构建验证规则

AI 运行构建命令后，**必须执行对应的 daemon/进程清理命令**。原因：命令行构建工具的守护进程可能持有文件锁，导致 IDE 构建冲突。例如：
- Android/Gradle：`gradlew --stop`
- 其他构建工具：根据实际情况清理

## 禁止行为

- 禁止跳过指南写代码
- 禁止回复不加签名
- 禁止签名列未读文档
- 禁止通读源码
- 禁止修改外部 Skill
- 禁止新增功能不更新文档
- 禁止删除功能不清底层代码和文档
- 禁止凭模糊记忆编写不熟悉的 API/框架代码，必须先搜索确认
- 禁止照搬外部搜索到的代码片段，必须适配项目架构后重写
- 禁止使用低质量信息源（复制粘贴站、无出处代码、未标明版本的教程）
- 禁止完成代码修改后跳过自审直接回复用户

## 冲突优先级

1. 用户最新要求
2. `PROJECT_GUIDE.md`
3. `GUIDE_*.md`
4. `GUIDE_USAGE_MATRIX.md`
5. `AI_WRITTEN_SKILLS/*.md`
6. 外部 Skill
```

### 4.3 AGENTS.md 模板

### 4.5 GUIDE_USAGE_MATRIX.md 模板

**重要说明**：这是所有模板中**最长也是最重要的**一个。绮梦影库的矩阵有 400+ 行，覆盖了 15+ 个模块。

**触发词必须写双列表（正式词 + 大白话）**。只写专业术语（"UI美化"、"性能优化"）是不够的——用户会说"这个界面改一下"、"感觉有点卡"、"推荐不太行"、"颜色不好看"。如果矩阵只覆盖正式词，这些大白话请求就不会命中任何指南。

新项目按规模估算：
- 小型项目（<5 模块）：每个模块约需正式词 5-8 个 + 大白话 8-15 个
- 中型项目（5-10 模块）：每个模块约需正式词 8-12 个 + 大白话 15-25 个
- 大型项目（10+ 模块）：每个模块约需正式词 10-15 个 + 大白话 20-30 个，并追加复合触发示例表和最终检查

**大白话词汇编写原则**：
- 覆盖情绪化表达：丑、太慢了、烦、不对、乱七八糟、不好看
- 覆盖感官描述：太大、太小、看不清、太暗、刺眼、卡
- 覆盖动作描述：去掉、加个、换一下、改改、隐藏
- 覆盖模糊指代：这个页面、那个界面、上面那一行、底部按钮
- 覆盖疑问句：怎么没了、为什么不对、能不能改

## 五、AI 自举操作流程

### 核心原则：AI 必须主动提问，不能等用户主动交代

用户说"帮我做一个图片管理器"时，AI 不能直接开始写代码。它应该像一个项目经理一样**主动追问**，直到收集齐足够信息才开始生成文档。大部分用户不知道项目需要准备什么，AI 必须引导。

---

### 第一步：主动提问，收集信息（AI 主导的交互式对话）

根据用户的一句话需求，AI 按以下顺序追问。每次只问 1-2 个问题，不要一口气全扔出来。

#### 轮次 1：确认基础信息

```
用户说："我想做一个图片管理软件"

AI 问：
"好的。先确认几个基础信息：
1. 这个软件运行在什么平台上？手机 App、电脑桌面软件、还是网页？
2. 你准备用什么技术栈？（如果不确定，我可以根据平台推荐）"
```

#### 轮次 2：确认架构和核心功能

```
用户回答后，AI 问：
"明白了。接下来：
1. 这个软件需要保存用户数据吗？（比如收藏记录、浏览历史、设置偏好）
2. 核心功能有哪些？你目前能想到的几个就行，不完整也没关系——我会补充常见功能。"
```

#### 轮次 3：AI 根据平台主动补充追问

```
用户回答后，AI 分析用户需求并追问用户可能遗漏的点：

【如果用户没提到数据库但提到了数据保存】
"你说要保存收藏记录——需要数据库来存储这些数据吗？我建议用 XXX，你看可以吗？"

【如果用户没提到主题/外观】
"你需要多套主题切换吗（比如浅色/深色模式）？如果是，UI 颜色需要统一管理。"

【如果用户没提到备份/导出】
"用户的数据需要支持导出备份吗？比如换设备时能把收藏和历史迁移过去。"

【如果用户没提到兼容性】
"需要支持哪些版本的系统？只做最新版还是兼容旧版？"

【如果用户提到图片/视频】
"图片和视频是只读展示，还是允许编辑？这个很重要，只读的话原文件完全不能动。"
```

#### 轮次 4：确认约束和边界

```
AI 追问：
"最后确认几件事：
1. 有没有绝对不能做的操作？（比如不能删原文件、不能联网）
2. 有没有外部 SDK 或第三方框架需要引入？
3. 这个项目是你一个人维护，还是可能多个 AI 或程序员轮换开发？"
```

### 第二步：AI 根据收集的信息推断文档清单

不需要用户确认以下细节，AI 自己根据决策树推断：

```
AI 内部处理：
- 平台 = Android → 需要 ANDROID_VIEW_SKILL, ANDROID_ARCHITECTURE_SKILL
- 数据库 = Room → 需要 ANDROID_ROOM_SKILL, GUIDE_DATA, GUIDE_BACKUP
- 有 UI = 多页面 → 需要 GUIDE_UI
- 有多主题 → 需要 GUIDE_THEME
- 有媒体 = 图片视频 → 需要 ANDROID_MEDIA_SKILL, GUIDE_SCAN
- 用户没提但项目需要 → AI 主动创建

生成清单后，AI 对用户说：
"根据你的需求，我准备创建这些文档（共 X 个文件）：
根目录 - README.md, AI_README_FIRST.md, AGENTS.md
项目总览 - docs/PROJECT_GUIDE.md
模块指南 - docs/GUIDE_UI.md, GUIDE_DATA.md, ...
通用规范 - docs/AI_WRITTEN_SKILLS/...
可以开始生成吗？或者你有要调整的？"
```

### 第三步：创建目录和文件

```
mkdir docs
mkdir docs/AI_WRITTEN_SKILLS
```

按 P0→P1→P2→P3 优先级批量生成。

| 优先级 | 文件 | 说明 |
|---|---|---|
| P0 | `AI_README_FIRST.md` | 几乎无项目特定内容，模板直填 |
| P0 | `AGENTS.md` | 几乎无项目特定内容，模板直填 |
| P0 | `README.md` | 填写名称、技术栈、功能 |
| P1 | `PROJECT_GUIDE.md` | 需用户确认定位、基准 |
| P1 | `GUIDE_README.md` | 根据模块列表生成导航 |
| P2 | `GUIDE_USAGE_MATRIX.md` | 为每个模块写触发词，**最耗时** |
| P2 | `DEVELOPMENT_PLAN.md` | 列出阶段和初始待办 |
| P3 | 项目 GUIDE_*（每个模块一个） | 用模板逐个生成 |
| P3 | AI_WRITTEN_SKILLS（按技术栈） | 根据需要创建 |

### 第四步：验证

- 检查 `AI_README_FIRST.md` 的读取顺序是否与实际文件对应
- 检查 `GUIDE_USAGE_MATRIX.md` 的引用路径是否全部有效
- 检查所有文档末尾有无 `> 最后更新：[日期]` 时间戳
- 检查模块索引和矩阵的交叉引用是否一致

### 第五步：向用户报告

```
AI 最后说：
"文档体系已生成，包含 [X] 个文件。你现在可以：
1. 打开 docs/GUIDE_README.md 看《你想做什么？》快速导航
2. 告诉我你想开始哪个功能，我根据文档直接开发
3. 如果有遗漏的模块，告诉我，我按新增流程补充"
```

## 六、日常维护规则（通用）

1. **改代码前**：走矩阵查指南 → 读了再动手
2. **改代码后**：更新对应文档 → 回复加签名
3. **加新模块**：按 7 步流程 → 不能成为无文档功能
4. **删功能**：逐层清代码 → 逐文档清描述
5. **AI 交班**：代码和文档同一 commit → commit message 注明改了哪些文档
6. **文档修正**：发现文档与代码不一致 → 先确认 → 再统一 → 不改错的
7. **工具调用**：遇到不熟悉的 API/框架/兼容性问题 → 先搜索确认再写代码 → 外部代码仅参考不照搬
8. **代码自审**：完成修改后 → 走自审清单（逻辑/线程/性能/一致性/副作用/文档）→ 发现问题先修复再回复

## 七、通用编码规范

以下规范适用于任何项目，应在 `GUIDE_CODE_MAINTENANCE.md` 或等效文档中记录。

### 代码阅读策略

**Grep + 精准读取**是行业验证的靠谱策略（Claude Code、OpenAI Codex CLI、Cursor 均采用）：

- 代码搜索关键词 95% 是标识符（类名36%、方法名41%、变量名18%），标识符是精确锚点
- 单轮 Grep 检索效果超过 embedding RAG 基线（GrepRAG, ISSTA '26）
- 搜索模式：Grep 定位行号 → offset/limit 精准读取 → 调整关键词 → 再搜
- 优势：零索引、零维护；trade-off：更多搜索轮次换取简洁性

### 命名规范

| 类型 | 规则 | 示例 |
|------|------|------|
| 类名 | 大驼峰 | `MediaFileEntity`、`HomeFragment` |
| 方法名 | 小驼峰，动词开头 | `recordView`、`deleteOrphanAuthors` |
| 变量名 | 小驼峰 | `isPregenerating`、`cachedLikeCounts` |
| 布尔变量 | is/has/can 前缀 | `isPregenerating`、`hasMore` |
| 常量 | 全大写下划线 | `MAX_LOG_SIZE`、`DATABASE_NAME` |
| 布局 ID | 蛇形小写，类型前缀 | `rvHomeGrid`、`tvTitle`、`btnScan` |

### 方法规范

采用三档警戒线（弹性规范，按功能实际需要决定代码规模）：

| 指标 | 绿色（常规） | 黄色（警戒，需备案） | 红色（需重构） |
|---|---|---|---|
| 单方法行数 | ≤80 | 80-120（需 KDoc 说明合理理由） | >120 |
| 类方法数（ViewModel） | ≤25 | 25-35 | >35 |
| 圈复杂度 | ≤10 | 10-20 | >20 |
| 嵌套深度 | ≤3 | 3-4 | >4 |
| 方法参数 | ≤5 | 5-8 | >8 |

- 方法参数超过 5 个则封装为数据类
- 嵌套超过 3 层则提取方法或使用 early return
- 黄色区需在 KDoc 注释说明"为何超长合理"（数据汇总/手势回调/扫描编排等合理复杂度）
- 红色区必须重构或显式备案技术原因

### 注释规范

- 关键业务逻辑必须添加行内注释
- 公共方法添加 KDoc（参数说明、返回值说明）
- 禁止无意义注释
- 注释语言与项目文档一致

### 空安全规范

- 优先使用 `?.` 和 `?:` 处理可空类型
- 禁止 `!!` 强制解包（除非在确认非空的初始化阶段）
- Fragment 中访问 binding 必须通过安全调用
- 方法返回可空类型时，在文档中说明何时返回 null

### 架构解耦规范

#### ViewModel 职责边界
- 单个 ViewModel 方法数不超过 25 个
- 超过时按功能拆分为 UseCase
- ViewModel 只做调度，不包含业务计算逻辑

#### Fragment 解耦
- Fragment 间禁止直接调用方法，应通过接口或共享 ViewModel 通信
- Fragment 不应持有其他 Fragment 的引用

#### 数据层解耦
- Repository 接口按功能域拆分查询方法
- DAO 方法粒度：一个方法只做一件事
- SQL 只在 DAO 中，Repository 不写 SQL

#### UI 与逻辑分离
- Fragment 只做 UI 绑定和事件分发
- 数据转换在 ViewModel 或 UseCase 中完成
- 筛选/排序/分组逻辑集中在工具类

### 代码质量指标

核心理念：**按功能实际需要决定代码规模，而非压行数**。行数只是气味信号，真正决定是否需要重构的是圈复杂度、模块内聚度、重复代码率、可测试性。

| 指标 | 绿色区目标 | 检查方式 |
|---|---|---|
| 单方法行数 | ≤80 行（黄色 80-120 需备案） | 静态分析 |
| 类/Fragment 总行数 | ≤600（黄色 600-1000 需备案） | 静态分析 |
| 方法数（ViewModel） | ≤25（黄色 25-35 需备案） | 静态分析 |
| 圈复杂度 | ≤10（黄色 10-20 需备案） | 静态分析 |
| 嵌套深度 | ≤3 层 | 静态分析 |
| 测试覆盖率 | > 60%（纯逻辑类 > 80%） | 测试框架 |
